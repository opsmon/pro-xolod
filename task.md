# Разработка сайта компании «Про-Холод»

Создай полноценный современный сайт-визитку для компании **«Про-Холод»** из Комсомольска-на-Амуре.

Сайт должен рассказывать об услугах компании, показывать ориентировочные цены, вызывать доверие и направлять посетителя к обращению во ВКонтакте, по телефону или через другой доступный канал связи.

Карточка компании в 2ГИС:

```text
https://2gis.ru/komsomolsk-on-amur/firm/70000001084062702
```

В проекте будет приложен логотип компании. Используй его как основу визуального стиля.

Сайт должен быть опубликован на **GitHub Pages**, чтобы готовую демо-версию можно было отправить клиенту по публичной ссылке для просмотра и согласования.

---

# 1. Основная задача

Разработай продающий, но ненавязчивый одностраничный сайт для локальной компании, которая занимается кондиционерами и климатическим оборудованием.

Пользователь должен за несколько секунд понять:

* чем занимается компания;
* в каком городе она работает;
* какие услуги оказывает;
* сколько примерно стоят услуги;
* почему компании можно доверять;
* как быстро связаться;
* как заказать расчёт стоимости.

Главное целевое действие:

> Написать компании и получить расчёт стоимости.

Не превращай сайт в сложный корпоративный портал. Это простой, понятный и визуально качественный лендинг.

# 8. Демо-плашка

Поскольку сайт сначала будет показываться клиенту как демонстрационная версия, добавь в верхней части страницы небольшую ненавязчивую плашку:

> Демо-версия сайта. Информация, контакты и цены находятся на согласовании.

Плашка должна:

* соответствовать дизайну сайта;
* быть компактной;
* хорошо выглядеть на телефоне;
* не мешать навигации;
* включаться и отключаться через конфигурацию.

Пример:

```ts
export const siteConfig = {
  demoMode: true,
  demoMessage:
    "Демо-версия сайта. Информация, контакты и цены находятся на согласовании.",
};
```

После утверждения сайта:

```ts
demoMode: false
```

При отключении режима плашка полностью скрывается.

Создай блок с преимуществами.

Объясняем, из чего складывается цена, и согласовываем работы до начала монтажа.

Обслуживание офисов, магазинов и других коммерческих помещений.

# 15. Как проходит работа

Создай последовательность из четырёх шагов.

### 1. Заявка

Клиент пишет во ВКонтакте, мессенджер или звонит.

### 2. Уточнение деталей

Специалист узнаёт площадь помещения, особенности установки и пожелания клиента.

### 3. Расчёт

Компания рассчитывает стоимость оборудования и монтажа.

### 4. Выполнение работ

Мастер приезжает в согласованное время, выполняет работу и проверяет оборудование.

На десктопе шаги можно показать горизонтально.

На телефоне — вертикально.

# 22. Счётчик посещений

Добавь в правом нижнем углу компактный виджет статистики посещений.

Он должен показывать:

* посещений сегодня;
* посещений за текущий месяц.

Пример:

```text
Посещения

Сегодня          24
За месяц        618
```

## Внешний вид

Виджет должен быть небольшим и ненавязчивым:

* светлый полупрозрачный фон;
* `backdrop-blur`;
* тонкая граница;
* мягкая тень;
* скругление 16–20 px;
* небольшая иконка статистики;
* спокойная типографика;
* маленький фирменный градиентный акцент;
* ширина примерно 160–210 px.

Расположение на десктопе:

```css
position: fixed;
right: 16px;
bottom: 16px;
z-index: 40;
```

На мобильном устройстве:

* поднимать виджет выше мобильной панели;
* уменьшить размер;
* при необходимости сделать сворачиваемым;
* не перекрывать кнопки и контент.

Добавь кнопку сворачивания.

В свёрнутом состоянии можно оставить только небольшую иконку с подписью:

> Статистика

Состояние сворачивания можно запоминать в `localStorage`.

## Техническая реализация

GitHub Pages не имеет собственного backend.

Нельзя использовать `localStorage` как основной общий счётчик, поскольку такой вариант считает посещения отдельно для каждого браузера.

Для рабочего счётчика подготовить интеграцию с внешним API.

Предпочтительная архитектура:

* сайт на GitHub Pages;
* Cloudflare Worker как API;
* Cloudflare KV, D1 или другое простое serverless-хранилище.

Допустимые альтернативы:

* Supabase;
* Firebase;
* Umami API;
* GoatCounter;
* собственный внешний endpoint.

Создай отдельный компонент:

```text
src/components/VisitCounter.tsx
```

Конфигурация:

```ts
export const analyticsConfig = {
  enabled: true,
  counterVisible: true,
  demoMode: true,

  endpoint: import.meta.env.VITE_VISITS_API_URL,

  refreshInterval: 60000,

  demoToday: 24,
  demoMonth: 618,
};
```

В рабочем режиме API должен возвращать:

```json
{
  "today": 24,
  "month": 618
}
```

Сайт должен корректно обрабатывать:

* загрузку;
* успешный ответ;
* отсутствие endpoint;
* сетевую ошибку;
* неправильный ответ;
* временную недоступность сервиса.

При ошибке:

* не ломать страницу;
* не показывать `NaN`;
* не показывать `undefined`;
* не генерировать случайные значения;
* либо скрыть виджет;
* либо показать нейтральный текст «Статистика временно недоступна».

## Учёт посещений

Не увеличивай счётчик при каждом рендере компонента.

Одно посещение учитывать:

* один раз за сессию;
* либо один раз в установленный промежуток;
* либо один раз в день для анонимного идентификатора.

Для защиты от повторного вызова можно использовать:

```ts
const VISIT_SESSION_KEY = "pro-holod-visit-counted";
```

`sessionStorage` или `localStorage` используется только для защиты от повторной отправки, но не как основное хранилище общей статистики.

## Приватность

Не собирать:

* ФИО;
* телефон;
* точную геолокацию;
* открытый IP-адрес;
* fingerprint;
* историю посещённых сайтов;
* другие лишние персональные данные.

Если внешний сервис обрабатывает IP или cookies, это необходимо указать в политике конфиденциальности.

## Демо-режим счётчика

Для показа клиенту разрешается использовать демонстрационные значения:

```ts
demoMode: true,
demoToday: 24,
demoMonth: 618
```

При этом рядом с данными должна быть небольшая подпись:

> Демо-данные

После подключения рабочего API:

```ts
demoMode: false
```

Не выдавай демонстрационные цифры за настоящую статистику.
